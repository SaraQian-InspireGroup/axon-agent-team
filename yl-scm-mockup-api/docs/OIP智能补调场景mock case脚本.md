# OIP 智能补调场景 Mock Case 脚本

> 用途：面向 FDE 的智能体建设与本体建模输入稿  
> 视角：补调经理  
> 场景边界：奶粉线下渠道分仓补调，覆盖日前例行补调与事件驱动临时补调  
> 参考材料：`场景reference材料/OIP智能补调业务场景-详细纪要.md`、`场景reference材料/奶粉分仓补货数据库表字段.xlsx`

---

## 0. 演示目标

本 mock case 用来说明：在现有 OIP 逻辑中，补调经理每天 9 点查看 Dashboard 异常项后，需要人工判断规则、修正策略、确认分配量并下发补货单。AgentOS 的目标不是替代 OIP 的基础数据和算法，而是在 OIP 之上形成“业务目标驱动的智能决策层”：

1. 自动识别异常项，解释异常原因。
2. 自动按业务规则和约束生成补调建议。
3. 自动输出多方案、决策依据和风险提示。
4. 人只做确认、修正或驳回。
5. 将人工反馈沉淀为后续可复用的规则、经验和本体关系。

本稿设计两条脚本：


| 脚本       | 类型          | 核心价值                                            | 人在环内位置         |
| -------- | ----------- | ----------------------------------------------- | -------------- |
| Script 1 | 9 点例行日前补调   | Agent 代替补调经理完成 Dashboard 异常分析、正向补货、横向调拨和补货单草案生成 | 经理确认最终下发       |
| Script 2 | 上下游事件驱动临时补调 | Agent 识别工厂、仓储、履约端事件，判断是否需要打破 routine 并发起临时补调    | 经理确认临时调整或补货单变更 |

## 1. 业务对象与数据口径



### 1.1 关键业务对象


| 对象      | 说明                   | 典型字段                                                                                |
| ------- | -------------------- | ----------------------------------------------------------------------------------- |
| 产品 SKU  | 补调的最小品项对象            | `product_code`、`product_name`、`brand`、`weight`、`volume`、`pro_series`                |
| 仓库      | 基地仓或销售仓              | `site_code`、`site_name`、`site_type`、`business_code`                                 |
| 销售计划    | 月度和次月计划              | `plan_num`、`next_plan_num`、`avg_plan_num`、`next_avg_plan_num`                       |
| 现货库存    | 仓库现有库存，含批次日期         | `store_num`、`actual_num`、`produce_date`、`status`                                    |
| 在途库存    | 已发未入库或已下发未发货量        | `store_transit`、`issued_not_dispatched`、`trans_order_not_dispatched`                |
| 实际销量/订单 | 当月订单、已出库、未发货         | `sell_num`、`out_put_num`、`available_quantity`、`unshipped_orders`                    |
| 监控报表    | OIP Dashboard 异常判断基础 | `order_gap`、`ship_gap`、`order_completion_rate`、`from_stock_rate_before`             |
| 正向调拨建议  | 基地仓到销售仓的分配方案         | `from_site_code`、`to_site_code`、`trans_num_jh`、`trans_num`、`reason`                 |
| 横向调拨建议  | 销售仓到销售仓的调拨方案         | `from_big_date_num`、`to_big_date_num`、`from_stock_rate_after`、`to_stock_rate_after` |
| 补货单     | 经理确认后下发给分仓/物流执行      | `push_num`、`push_time`、`push_user`、`remark`                                         |
| 履约事件    | 出库、运输、入库和异常事件        | 车牌号、定位时间、位置、速度、WMS 状态、TMS 状态                                                        |




### 1.2 关键计算口径

```text
发货缺口 = 现有库存 + 在途库存 - 未发订单

月度供应 = 现库 + 当月累出 + 生产计划

月度需求 = 销售计划

生产备货率 = (现货库存 + 在途库存 + 本月已出库) / 销售计划

订单完成率 = (当月未发货量 + 本月累出) / 销售计划
```



### 1.3 规则基线


| 库存状态 | 全国供应 vs 月度需求 | 中旬目标逻辑示例                              |
| ---- | ------------ | ------------------------------------- |
| 充足   | 供应 > 需求 +5%  | 标准进度 50%；订单进度超标准时，在订单进度基础上 +20%，不超过计划 |
| 持平   | 供应在需求 ±5% 内  | 中旬按订单进度 +20%，不超过计划                    |
| 不足   | 供应 < 需求 -5%  | 优先满足订单，不主动超前备货                        |


优先级原则：

1. 服务水平和不断货优先。
2. 时效优先，其次考虑成本。
3. 就近调拨优先。
4. 有临期或大日期库存时，优先通过横向调拨消化老批次。
5. 偏远仓、旺季、无计划有订单、定制/电商/试饮装等情况需要特殊规则。

### 2.3 AgentOS 自动分析过程



#### Step 1：识别业务意图

补调经理打开 Dashboard 后，AgentOS 自动识别当前任务：

```text
任务类型：日前例行补调
分析范围：线下渠道，奶粉品类，全国销售仓
分析粒度：SKU x 销售仓 x 基地仓/销售仓来源
目标：将风险仓调整至目标备货率，同时控制老日期库存风险
```



#### Step 2：判断全国库存状态

Agent 读取 `销售计划`、`现货库存`、`在途库存`、`实际销量` 与 OIP 监控报表：

```text
SKU NF-JGZ-3-900 全国供应 = 107,000 件
SKU NF-JGZ-3-900 全国需求 = 100,000 件
判断：供应高于需求 7%，库存状态 = 充足

SKU NF-XH-800 全国供应 = 82,000 件
SKU NF-XH-800 全国需求 = 80,000 件
判断：供应高于需求 2.5%，库存状态 = 持平
```



#### Step 3：异常 A 的正向补货推理

```text
北京销售仓当前生产备货率
= (现货库存 1,200 + 在途 300 + 本月已出库 3,000) / 销售计划 10,000
= 45%

北京销售仓订单完成率
= (当月未发 4,200 + 本月已出库 3,000) / 销售计划 10,000
= 72%

当前为中旬，标准进度为 50%。
该 SKU 全国库存充足，且北京订单进度 72% > 标准进度 50%。
目标备货率 = 订单完成率 72% + 20% = 92%，不超过 100%。

目标库存口径 = 10,000 x 92% = 9,200 件
当前口径库存 = 1,200 + 300 + 3,000 = 4,500 件
建议补货量 = 9,200 - 4,500 = 4,700 件
```

Agent 选择发货仓：

```text
天津基地仓可发 8,000 件，T+1 到仓。
武汉基地仓可发 12,000 件，T+3 到仓。
因当前首要目标是保障北京仓订单履约，且天津基地仓可发量覆盖 4,700 件，所以推荐 JD-TJ → XS-BJ。
```

输出建议：


| 动作   | 发货仓         | 收货仓         | 建议量     | 原因                                     |
| ---- | ----------- | ----------- | ------- | -------------------------------------- |
| 正向调拨 | JD-TJ 天津基地仓 | XS-BJ 北京销售仓 | 4,700 件 | 北京订单完成率 72%，备货率仅 45%，需要提升至 92%；天津仓时效最优 |




#### Step 4：异常 B 的横向调拨推理

```text
南京销售仓生产备货率 = (8,000 + 0 + 1,500) / 6,000 = 158%
南京大日期库存 = 3,200 件，存在积压和效期风险。

上海销售仓生产备货率 = (2,000 + 500 + 4,500) / 12,000 = 58%
上海订单完成率 = (4,000 + 4,500) / 12,000 = 71%

当前为中旬，上海订单进度高于标准进度。
目标备货率按 71% + 20% = 91% 计算。
目标口径库存 = 12,000 x 91% = 10,920 件
当前口径库存 = 2,000 + 500 + 4,500 = 7,000 件
理论缺口 = 3,920 件
```

Agent 判断：

1. 南京库存明显高于计划，且有 3,200 件大日期库存。
2. 上海存在订单缺口，但同 SKU 在南京有可消化老日期库存。
3. 优先用横向调拨消化南京大日期，再用基地仓补足剩余缺口。

输出建议：


| 动作   | 发货仓         | 收货仓         | 建议量     | 原因                   |
| ---- | ----------- | ----------- | ------- | -------------------- |
| 横向调拨 | XS-NJ 南京销售仓 | XS-SH 上海销售仓 | 2,500 件 | 优先消化南京大日期库存，同时缓解上海缺口 |
| 正向补货 | JD-TJ 天津基地仓 | XS-SH 上海销售仓 | 1,400 件 | 横调后上海仍低于目标备货率，补足剩余缺口 |


调拨后效果：


| 仓库    | 调整前备货率 | 调整后备货率 | 风险变化                   |
| ----- | ------ | ------ | ---------------------- |
| 南京销售仓 | 158%   | 117%   | 库存过高风险下降，大日期消化 2,500 件 |
| 上海销售仓 | 58%    | 90.5%  | 缺货风险基本解除               |




#### Step 5：生成经理确认卡片

AgentOS 向补调经理展示：

```text
今日共识别 2 个高优先级异常、1 个中优先级异常。

建议动作：
1. JD-TJ → XS-BJ，NF-JGZ-3-900，正向调拨 4,700 件。
2. XS-NJ → XS-SH，NF-XH-800，横向调拨 2,500 件。
3. JD-TJ → XS-SH，NF-XH-800，正向调拨 1,400 件。

预计效果：
- 北京销售仓备货率从 45% 提升至 92%。
- 上海销售仓备货率从 58% 提升至 90.5%。
- 南京销售仓大日期库存减少 2,500 件。

需要人工确认：
- 是否接受南京向上海横向调拨 2,500 件？
- 上海正向补货 1,400 件是否需要凑车加量？
- 北京补货是否直接下发，还是等待今日 11 点 WMS 可发量刷新？
```

## 3. Script 2：上下游事件驱动的临时补调



### 3.1 场景设定


| 项目             | Mock 值                                  |
| -------------- | --------------------------------------- |
| 时间             | T 日 10:30，9 点例行补调方案已生成但尚未全部执行           |
| 触发方式           | AgentOS 监听工厂、WMS、TMS、履约中台事件             |
| 业务目标           | 当上游供给或下游履约发生突发变化时，及时判断是否需要临时补调          |
| 与 Script 1 的差异 | 不是固定 9 点 Dashboard 批处理，而是事件流触发的实时/准实时重算 |




### 3.2 事件 1：工厂入库延期，原补货方案失效



#### 事件输入

```text
事件时间：T 日 10:30
事件来源：工厂排产 / WMS 入库事件
事件类型：生产入库延期
产品：NF-JGZ-3-900 金领冠珍护 3 段 900g
基地仓：JD-TJ 天津基地仓
原计划：T 日 18:00 入库 5,000 件
新计划：T+2 日 18:00 入库 5,000 件
影响：天津基地仓今日可发量由 8,000 件降至 3,200 件
```



#### AgentOS 事件识别

Agent 自动判断该事件影响上午已生成的北京补货建议：

```text
已确认但未下发方案：JD-TJ → XS-BJ，4,700 件
天津基地仓最新可发量：3,200 件
缺口：1,500 件

若仍从天津发 4,700 件，则违反可发量约束。
若只发 3,200 件，北京销售仓备货率只能提升至 77%，低于目标 92%。
需要临时重算来源仓。
```



#### AgentOS 临时补调建议

备选方案：


| 方案  | 调拨路径          | 数量      | 到仓  | 优点         | 风险           |
| --- | ------------- | ------- | --- | ---------- | ------------ |
| A   | JD-TJ → XS-BJ | 3,200 件 | T+1 | 最近，最快      | 不能完全达标       |
| B   | JD-WH → XS-BJ | 1,500 件 | T+3 | 可补足缺口      | 到仓较晚         |
| C   | XS-TJ → XS-BJ | 1,200 件 | T+1 | 横向快，缓解短期履约 | 需确认天津销售仓不被拉低 |


Agent 推荐组合方案：


| 动作     | 路径            | 数量      | 原因               |
| ------ | ------------- | ------- | ---------------- |
| 正向补货   | JD-TJ → XS-BJ | 3,200 件 | 天津基地仓最新可发量上限     |
| 横向临时调拨 | XS-TJ → XS-BJ | 1,200 件 | T+1 到仓，先保障北京未发订单 |
| 正向补货   | JD-WH → XS-BJ | 300 件   | 补足目标备货率剩余缺口      |


Agent 解释：

```text
由于天津基地仓入库延期，原方案 4,700 件不可执行。
北京未发订单 4,200 件，短期服务水平风险高。
天津销售仓当前备货率 118%，横调 1,200 件后仍为 103%，不触发自身风险。
因此建议使用“天津基地仓可发量 + 天津销售仓横调 + 武汉基地仓补尾差”的组合方案。
```

### 3.3 事件 2：履约端大客户追加订单，触发临时补货



#### 事件输入

```text
事件时间：T 日 14:20
事件来源：履约中台 / OMS
事件类型：大客户追加订单
产品：NF-XH-800 欣活成人奶粉 800g
销售仓：XS-SH 上海销售仓
订单量：2,800 件
要求：T+2 前出库
渠道：重点母婴渠道
```

事件发生前，上午方案已将上海备货率提升到约 90.5%。追加订单后：

```text
上海销售仓新增未发订单 = 2,800 件
调整后未发订单 = 4,000 + 2,800 = 6,800 件
调整后订单完成率 = (6,800 + 4,500) / 12,000 = 94.2%
原补调后可用库存口径 = 2,000 + 500 + 2,500 + 1,600 + 4,500 = 11,100 件
目标备货率 = min(94.2% + 20%, 100%) = 100%
目标口径库存 = 12,000 件
新增临时缺口 = 12,000 - 11,100 = 900 件
```



#### AgentOS 临时补调建议

Agent 判断：

1. 订单来自重点渠道，履约时效 T+2，优先级高。
2. 上海上午已安排南京横调 2,500 件和天津正向补货 1,600 件。
3. 追加订单导致目标备货率上调至 100%，仍缺 900 件。
4. 南京可继续横调的大日期库存上限为 700 件，超过 700 件会影响南京自身安全边界。
5. 杭州销售仓同 SKU 库存健康，可横调 300 件，T+1 到上海。

推荐：


| 动作     | 路径            | 数量    | 原因                               |
| ------ | ------------- | ----- | -------------------------------- |
| 追加横向调拨 | XS-NJ → XS-SH | 600 件 | 继续消化南京剩余可调大日期库存，并保留 100 件安全缓冲    |
| 追加横向调拨 | XS-HZ → XS-SH | 300 件 | 杭州到上海 T+1，可覆盖订单时效                |
| 风险提示   | 无             | -     | 追加后上海达到 100% 备货率，但需物流确认 T+2 发运能力 |


Agent 同时提示：

```text
本次是 routine 以外的事件驱动临时补调。
建议不等待次日 0 点批处理，直接进入经理确认。
如物流运力不足，应触发履约协同 Agent 查询可用车辆或凑车方案。
```

### 4.3 Rule


| Rule                            | 内容                                          |
| ------------------------------- | ------------------------------------------- |

| `human_approval_rule`           | 所有补货单下发前必须由补调经理确认                           |
| `event_override_rule`           | 工厂、订单、WMS、TMS 事件影响已生成方案时，可触发临时重算并打破 routine |
